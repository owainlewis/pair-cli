# Agent Workspace CLI

## What

Build `pair`, a single-binary Go CLI for AI agents and humans to operate a PAIR workspace from a terminal or automation environment. The CLI should make PAIR's existing bearer-token HTTP API feel like a local project workspace: agents can list tasks, claim or update work, comment progress, publish markdown artifacts, and fetch or update documents without hand-writing `curl` requests.

## Context

The PAIR web app lives in the companion `owainlewis/pair` Rails repo. It is a Rails app for agent-native markdown writing and task management. The current API is defined in `app/controllers/api/v1/*`, with behavior covered by `test/integration/api/*`.

Relevant existing contracts:

- Authentication is `Authorization: Bearer <token>` for every `/api/v1/*` endpoint. Tokens are generated by `ApiToken.create_for_user!` with the `pair_` prefix, stored only as SHA-256 digests, and `last_used_at` is touched on each API request.
- JSON API errors use `{ "error": "<machine_code>", "message": "<human>" }`. Content endpoints override this and return `text/plain`.
- Resource IDs exposed to clients are opaque `public_token` strings, not database IDs.
- Documents are raw markdown. Metadata is JSON, but `GET /api/v1/documents/:id/content` and `PUT /api/v1/documents/:id/content` are the byte-oriented markdown path.
- Tasks use fixed statuses: `todo`, `doing`, `review`, `done`.
- Agents can create task comments and can link documents to tasks or collections by either existing document ID or a new markdown body.

This `pair-cli` repo is initialized with a README and Go `.gitignore`, but no Go source yet.

## Requirements

1. The project must produce a single executable named `pair`.
2. The CLI must run on macOS and Linux for both ARM64 and AMD64.
3. The CLI must support non-interactive agent workflows by accepting all required inputs via flags, stdin, environment variables, or config.
4. The CLI must authenticate to PAIR with a bearer token and never print the token in normal output, logs, or error messages.
5. The CLI must support configuring a base URL so it works with local development (`http://localhost:3000`) and production.
6. The CLI must expose task commands for list, show, create, update status, comment, link document, publish document-to-task, and delete.
7. The CLI must expose document commands for list, search/filter, show metadata, read markdown content, create, replace content, and delete.
8. The CLI must expose collection commands for list, show, create, rename, attach existing document, publish markdown into a collection, detach document, and delete.
9. The CLI must default to human-readable tables for terminal use and support `--json` for machine-readable agent use.
10. Commands that read or write markdown must support `--file <path>` and stdin via `-`.
11. Destructive commands must require explicit confirmation unless `--yes` is supplied.
12. API failures must preserve the server's status code and machine error code in a way scripts can inspect.
13. The CLI must include focused tests for command parsing, config resolution, HTTP request construction, response formatting, and error handling.

## Design

### CLI shape

Use `pair` as the binary and root command. Commands should be resource-oriented and close to the API names:

```text
pair auth status
pair config set base-url http://localhost:3000
pair config set token pair_...

pair tasks list [--status todo|doing|review|done] [--json]
pair tasks show <task-id> [--json]
pair tasks create --title <title> [--description <md>] [--status todo|doing|review|done] [--json]
pair tasks status <task-id> <todo|doing|review|done> [--json]
pair tasks comment <task-id> (--body <text> | --file <path-or->) [--json]
pair tasks link-doc <task-id> <document-id> [--json]
pair tasks publish <task-id> (--file <path-or-> | --body <md>) [--tag <tag>...] [--json]
pair tasks unlink-doc <task-id> <document-id> [--yes]
pair tasks delete <task-id> [--yes]

pair docs list [--query <text>] [--tag <tag>] [--since <Nd|all>] [--json]
pair docs show <document-id> [--json]
pair docs read <document-id> [--output <path>]
pair docs create (--file <path-or-> | --body <md>) [--tag <tag>...] [--json]
pair docs replace <document-id> (--file <path-or-> | --body <md>)
pair docs delete <document-id> [--yes]

pair collections list [--json]
pair collections show <collection-id> [--json]
pair collections create --name <name> [--json]
pair collections rename <collection-id> <name> [--json]
pair collections link-doc <collection-id> <document-id> [--json]
pair collections publish <collection-id> (--file <path-or-> | --body <md>) [--tag <tag>...] [--json]
pair collections unlink-doc <collection-id> <document-id> [--yes]
pair collections delete <collection-id> [--yes]
```

`Assumption:` `pair` is the preferred binary name even though the repository is `pair-cli`. If that creates a naming conflict later, only the build artifact name needs to change.

### Configuration

Resolve configuration in this order:

1. Command flags: `--base-url`, `--token`.
2. Environment variables: `PAIR_BASE_URL`, `PAIR_TOKEN`.
3. Config file: `$XDG_CONFIG_HOME/pair/config.json`, falling back to `~/.config/pair/config.json`.

Config file shape:

```json
{
  "base_url": "https://usepair.ai",
  "token": "pair_example"
}
```

The token should be written with file mode `0600` on Unix-like systems. The first implementation can store the token in the config file; OS keychain integration is out of scope for v1.

### HTTP client

Create a small internal API package around `net/http`:

- `Client` owns `BaseURL`, `Token`, `HTTPClient`, and JSON/text helpers.
- JSON requests set `Content-Type: application/json` and `Accept: application/json`.
- Markdown content replacement sets `Content-Type: text/markdown; charset=utf-8`.
- Every authenticated request sets `Authorization: Bearer <token>`.
- 204 responses return nil data and are still treated as success.
- Non-2xx JSON errors decode into an `APIError` with status, code, and message.
- Non-2xx text errors decode into an `APIError` with status and message.

The client should not hide server behavior. For example, invalid task statuses are currently ignored by the Rails API during update, so the CLI should validate statuses locally before sending the request.

### Output

Default output should be terse and readable:

- Lists render stable tables with IDs first.
- Mutating commands print the updated or created resource summary.
- `docs read` writes raw markdown to stdout unless `--output` is provided.
- `--json` prints the server-equivalent JSON response plus only minimal CLI additions when necessary.
- Errors print one line to stderr and exit non-zero, for example: `error: 404 not_found: Resource not found`.

Exit codes:

- `0`: success.
- `1`: local usage, validation, config, or filesystem error.
- `2`: server/API error with a received HTTP status.

### Package layout

Use a small, conventional Go layout:

```text
cmd/pair/main.go
internal/cli
internal/config
internal/api
internal/output
```

Keep the API types boring and explicit: `Document`, `Task`, `TaskComment`, `Collection`, and `APIError`. Avoid generating code from the Rails app for v1; the API is small enough that hand-written structs are clearer.

### API mapping

Documents:

- `GET /api/v1/documents?q=&tag=&since=`
- `POST /api/v1/documents`
- `GET /api/v1/documents/:id`
- `GET /api/v1/documents/:id/content`
- `PUT /api/v1/documents/:id/content`
- `DELETE /api/v1/documents/:id`

Tasks:

- `GET /api/v1/tasks?status=`
- `POST /api/v1/tasks`
- `GET /api/v1/tasks/:id`
- `PATCH /api/v1/tasks/:id`
- `DELETE /api/v1/tasks/:id`
- `POST /api/v1/tasks/:task_id/comments`
- `POST /api/v1/tasks/:task_id/documents`
- `DELETE /api/v1/tasks/:task_id/documents/:id`

Collections:

- `GET /api/v1/collections`
- `POST /api/v1/collections`
- `GET /api/v1/collections/:id`
- `PATCH /api/v1/collections/:id`
- `DELETE /api/v1/collections/:id`
- `POST /api/v1/collections/:collection_id/documents`
- `DELETE /api/v1/collections/:collection_id/documents/:id`

## Decisions

1. **Use Go and ship one binary.** Alternative: Ruby, Node, or shell wrappers around `curl`. Go is the right fit because the install story is one executable, cross-compilation is straightforward, and the runtime dependency story is clean. Reversible only by rewriting the CLI.

2. **Use Cobra for command routing.** Alternative: Go's `flag` package. Cobra adds a dependency, but it gives nested commands, help text, shell completion support, and a familiar UX for resource-oriented CLIs. Reversible, but changing later would touch every command.

3. **Use the standard library `net/http` client.** Alternative: Resty or another HTTP client package. The API is simple, and `net/http` avoids an unnecessary runtime dependency. Reversible if retry, tracing, or middleware needs grow.

4. **Store v1 credentials in an XDG config file.** Alternative: OS keychains. A config file is portable, scriptable, and good enough for agents running in containers or remote shells. Reversible; keychain storage can be added later while preserving env var and flag precedence.

5. **Make `--json` the machine interface, not the default.** Alternative: default JSON everywhere. Humans will also use the CLI, and tables make board triage easier. Reversible command-by-command, but defaults should stay stable once users script against them.

6. **Validate task status locally.** Alternative: rely on server behavior. The current API ignores invalid statuses on PATCH, which is safe for the server but surprising in a CLI. Local validation gives clear feedback before a request. Reversible if the Rails API later changes to reject invalid statuses itself.

7. **Do not add a custom sync/cache layer in v1.** Alternative: local SQLite cache or workspace mirror. The PAIR API is already the source of truth, and v1 should stay thin. Reversible when offline or bulk workflows become real.

## Versions

- Go: target Go 1.26.x, with Go 1.26.3 as the current stable patch release verified from the official Go release history on 2026-05-27: https://go.dev/doc/devel/release
- Cobra: use the current stable `github.com/spf13/cobra` release at implementation time and pin it in `go.mod`.
- Test tooling: prefer Go's built-in `testing` package and `httptest`; add assertion helpers only if they materially improve readability.

## Invariants

- The CLI must never log or print bearer tokens, including in debug-style error output.
- `docs read` and `docs replace` must preserve markdown bytes as much as the OS and Go runtime allow; do not trim trailing newlines, normalize line endings, or parse markdown.
- Resource IDs passed to API endpoints must be URL path escaped.
- The default base URL must not silently point at production during tests.
- `--json` output must remain valid JSON on stdout, with errors on stderr.
- Destructive commands must not proceed without `--yes` when stdin is not interactive.

## Error Behavior

- Missing token: print a local config error with instructions to set `PAIR_TOKEN` or `pair config set token`.
- Missing base URL: use the default production URL once it is known; until then, require config and show an actionable error.
- Network failure: print the method, sanitized URL, and network error; exit `1`.
- API JSON error: print `status code + machine code + message`; exit `2`.
- API text error from content endpoints: print `status code + message`; exit `2`.
- Invalid stdin/file input: print the local file error; exit `1`.
- Confirmation declined: print nothing or a short `aborted`; exit `1`.

## Testing Strategy

- Unit tests for config precedence: flags over env vars over config file.
- Unit tests for token redaction in errors and formatted output.
- `httptest.Server` tests for each API package method to verify method, path, query params, auth header, content type, request body, and response parsing.
- CLI command tests that execute commands with in-memory stdout/stderr and fake API clients.
- Golden-ish tests for table output where useful, but prefer structural assertions for stability.
- Integration smoke test against a local PAIR server can be added later behind an explicit environment variable such as `PAIR_CLI_INTEGRATION=1`.
- Release build check should compile `cmd/pair` for `darwin/arm64`, `darwin/amd64`, `linux/arm64`, and `linux/amd64`.

## Out of Scope

- MCP server implementation.
- Local database, offline mode, background sync, or file watcher.
- Browser login, API token creation, or password authentication from the CLI.
- OS keychain integration.
- Shell completions beyond what Cobra can generate trivially.
- Windows support for v1.
- Managing public share toggles, because the current Rails API does not expose share create/destroy endpoints under `/api/v1`.
- GitHub project, issue, or PR integration.
